#The idea

Everybody has a business idea. Almost nobody has a cheap, fast way to find out whether it survives contact with other people. Asking friends produces politeness, not truth; building an MVP costs months. Idea Arena is the step in between: post the idea, let a crowd of strangers pick it apart, and watch an honest score emerge from votes and arguments.

Three design decisions make the feedback honest:

1. Every idea is a structured pitch, not a wall of text

An idea is submitted through fixed fields — the same questions an investor would ask:

Unique Value Proposition Problem Solution Competition Income strategy Exit strategy Video pitch

Structure forces clarity from the author — and gives the community something concrete to attack.

2. Feedback is aimed, not general

Comments carry a metric badge: a thread can target the exit strategy specifically, or the problem statement, or the idea as a whole. Instead of one noisy pile of replies, the author sees exactly which part of the pitch is weak. Replies inherit the badge of their parent, so each discussion stays on topic.

3. The clock creates urgency

An idea lives in the arena for exactly 14 days. While active, it competes in the feed for votes and discussion. When time runs out it leaves the feed, its score freezes, and it is ranked among the retired ideas — the top of that ranking is the Winners board. A deadline means voting momentum matters, the feed never silts up with zombies, and "winning" actually means something.

#How it works

  1. Sign up & pick an avatar Create an account, choose an avatar from the built-in collection, and you're in the arena.
  2. Post your idea Fill in the pitch: name, UVP, problem and solution are required; competition, income strategy, exit strategy and a video pitch link are optional ammunition.
  3. The community votes Reddit rules: one account, one vote per idea or comment. Press the same arrow again to take it back; press the other one to flip.
  4. Discussion drills into metrics Commenters tag their thread with the metric it targets. Threads nest infinitely, and every reply stays bound to its topic.
  5. Fourteen days on the clock Sort the feed by Top, New, Best (engagement), Controversial or Old while the battle runs — or search the arena by name, UVP and problem text.
  6. Winners are crowned After expiry the final score is frozen. The Winners view ranks all retired ideas; the top spot is the reigning champion.
  7. Take it further, privately Interested in an idea (or its author)? Send a chat request. If they accept, a private realtime conversation opens — the deal room after the pitch.

#Features

Reddit-style voting

Toggle & flip semantics, one vote per user, on both ideas and comments.

Metric-targeted threads

Comment threads aimed at a specific pitch field, with color-coded badges.

14-day idea lifecycle

Ideas expire out of the feed; scores freeze; winners are ranked forever.

Six feed sorts + search

Top, New, Best, Controversial, Old, Winners — plus free-text search.

Infinite scroll

The feed streams in pages of 10 as you scroll, no pagination buttons.

Chat with consent

Private chat opens only after a request is accepted — no unsolicited DMs.

Live notifications

Chat requests, acceptances and messages arrive instantly over WebSockets.

Avatars

Pick an identity from the built-in avatar set at registration, change it anytime.

#Architecture

CQRS with MediatR

Every use case is a small class pair: a Command or Query record describing intent, and a Handler executing it. Controllers stay one line long — they forward the request to MediatR and return the result. A validation behaviour runs FluentValidation rules on every command before any handler sees it, and a global exception filter translates domain exceptions (NotFound, Forbidden, Validation) into proper HTTP status codes.

Ports & adapters

Handlers depend on interfaces, never on technology: IApplicationDbContext hides EF Core, ICurrentUserService hides the auth cookie, IRealtimeNotifier hides SignalR. Swap PostgreSQL or the realtime transport tomorrow — the business layer doesn't change a line.

Authentication

Classic and boring on purpose: ASP.NET Identity issues an encrypted, HTTP-only cookie at login. The browser attaches it to every API call and to the WebSocket handshake, so one mechanism guards both worlds. Unauthenticated API calls get a clean 401 — no redirects, the SPA owns all navigation.

#Realtime, explained

Users never connect to each other — browsers can't accept incoming connections. Instead every signed-in user holds one persistent WebSocket to the server (SignalR, hub /hubs/chat), and the server routes between them like a switchboard:

User A Server User B │ │ │ │ POST /api/chat/…/messages │ ├────────────────────────►│ 1. validate (participant? │ │ │ request accepted?) │ │ │ 2. save message to PostgreSQL │ │ │ 3. look up B's live connections │ │ │ in the in-memory user map │ │ 4. HTTP 200 + DTO │ 5. push same DTO over WebSocket │ │◄────────────────────────┤──────────────────────────────────►│ │ (sender's UI updates) │ (recipient's UI updates)│

Design rules that keep it simple:

  • All actions go over HTTP. Sending a message is a normal POST with the full validation pipeline. The hub class is literally empty — the socket is a delivery door, not an entrance.
  • The socket only pushes server → client: two events, chatMessage and notification. It comes up at login and goes down at logout, app-wide — so the notification bell is live even while you browse the feed.
  • PostgreSQL is the source of truth. If the recipient is offline the push is a no-op; the message waits in the database and appears on their next fetch. The socket is an accelerator, never the storage.
  • Identity comes from the cookie. SignalR maps each connection to the user id claim, so handlers target a user (all their open tabs), not a raw socket.

#Run it locally

Prerequisites: .NET 8 SDK, Node 18+, and PostgreSQL (native install, or docker compose up -d with a .env file — see .env.example).

1 — Configure the database password (stored in user-secrets, never in the repo):

cd BusinessIdea
dotnet user-secrets set "ConnectionStrings:DefaultConnection" \
  "Host=localhost;Port=5432;Database=businessIdea;Username=postgres;Password=<yours>"

2 — Start the API (applies migrations automatically on startup):

cd BusinessIdea
dotnet run          # optionally: dotnet run -- seed  (mock data, then exits)

3 — Start the SPA (proxies /api and /hubs to the backend):

cd frontend
npm install
npx ng serve        # → http://localhost:4200
